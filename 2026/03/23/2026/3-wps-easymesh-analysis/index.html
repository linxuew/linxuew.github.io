WPS EasyMesh组网 & 一键桥接自适应

发表于2026-03-23|更新于2026-03-24|网络
|阅读量:

背景

在无线组网场景下，STA（客户端/下级节点）需要通过 WPS 完成入网，并自动判断上级 AP 是否支持 EasyMesh。若支持，则走 Multi-AP 组网流程成为 Mesh Agent；若不支持，则降级为普通桥接模式。本文基于一份真实的 pcapng 抓包，完整分析 WPS 握手过程、EasyMesh 探测机制。

一、WPS M1–M8 完整握手流程

WPS 握手分两个阶段：M1–M4 完成密钥协商与身份验证，M5–M8 完成凭证传递。

消息字段详解

消息 方向 关键字段
M1 STA → AP Version, UUID-E, MAC, EnrolleeNonce, Public Key (DH), Auth Type Flags, Encr Type Flags, Multi-AP Extension (0x1048, 若支持 EasyMesh)
M2 AP → STA RegistrarNonce, Public Key (DH), EncrSettings, Multi-AP Extension (回应 EasyMesh, 若支持)
M3 STA → AP RegistrarNonce, E-Hash1, E-Hash2 (PIN 前后半段哈希承诺)
M4 AP → STA EnrolleeNonce, R-Hash1, R-Hash2, Encrypted R-S1
M5 STA → AP EnrolleeNonce, Encrypted E-S1
M6 AP → STA EnrolleeNonce, Encrypted R-S2
M7 STA → AP EnrolleeNonce, Encrypted E-S2
M8 AP → STA Encrypted Settings（SSID + 密码 + 认证方式，EasyMesh 时额外含 Backhaul 配置）

密钥派生链

所有消息的加密与认证均基于同一条密钥链：

PIN 零知识证明机制

PIN 被切成前后两半，每半生成两个随机种子（E-S1/E-S2、R-S1/R-S2）：

  • M3/M4：先发哈希承诺（E-Hash、R-Hash）
  • M5–M7：再分批发明文种子，对端重新计算哈希对比

这种设计保证单条消息被截获时攻击者无法直接恢复 PIN，但也引出了著名的 Pixie Dust 漏洞（部分厂商 E-S1/E-S2 使用弱随机数）。

二、支持与不支持 EasyMesh 的 WPS 差异

不支持 EasyMesh（标准 WPS）

设备入网后不感知其他 AP，无缝漫游需依赖设备自身（802.11r/k/v，不保证生效）。

支持 EasyMesh（Multi-AP WPS）

核心区别：M1 中携带 Multi-AP Extension (0x1048)，M8 回传 Backhaul 凭证，握手结束后 Agent 通过 1905.1 消息向 Controller 注册，加入统一管控体系。

小结

本文从 WPS M1–M8 协议细节出发，结合真实抓包（普通 AP 不支持 EasyMesh 场景），完整梳理了 STA 自适应探测机制的四层判断逻辑。核心思路是：STA 主动声明 EasyMesh 能力，依据 AP 响应结果分叉行为，失败后清零标志自然回退。
